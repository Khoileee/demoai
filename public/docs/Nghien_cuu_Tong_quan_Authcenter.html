1. GIỚI THIỆU

1.1 Bối cảnh

Tổ chức hiện đang vận hành nhiều hệ thống nghiệp vụ nội bộ, mỗi hệ thống có cơ chế quản lý quyền truy cập riêng biệt. Điều này dẫn đến nhiều khó khăn trong công tác quản trị, bảo mật và trải nghiệm người dùng.

1.2 Phạm vi tài liệu

Tài liệu này trình bày:

  • Hiện trạng: Thực trạng các hệ thống phân quyền đang vận hành
  • Vấn đề: Các điểm nghẽn và thách thức gặp phải
  • Đề xuất: Giải pháp hệ thống phân quyền tập trung AuthCenter
  • Luồng nghiệp vụ: Cách thức hoạt động của hệ thống mới
  • Lộ trình: Kế hoạch triển khai theo giai đoạn

1.3 Các hệ thống trong phạm vi

STT Hệ thống Mô tả Số lượng user
1 SQLWF SQL Workflow - Quản lý truy vấn, bảng dữ liệu, job Rất nhiều
2 VDSD VDS Dashboard - Dashboard, báo cáo, chỉ tiêu Trung bình
3 Báo cáo VTM Báo cáo nghiệp vụ tích hợp Tableau Trung bình
4 AuthCenter Hệ thống phân quyền tập trung (đang phát triển) Mới

2. HIỆN TRẠNG

2.1 Kiến trúc phân quyền hiện tại

Hiện tại, tổ chức đã có hệ thống SSO (Single Sign-On) - tất cả nhân viên đăng nhập bằng chung một tài khoản của công ty. Tuy nhiên, mỗi hệ thống lại tự xây dựng và quản lý module phân quyền riêng biệt:

2.2 Đặc điểm từng hệ thống

2.2.1 SQLWF - Mô hình ACL (Access Control List)

Đặc điểm Mô tả
Cách thức phân quyền Cấp quyền trực tiếp cho từng user
Đơn vị quản lý Menu + Permission (khoảng 30 menu, 150+ quyền)
Ưu điểm Linh hoạt, cấp quyền chi tiết đến từng người
Nhược điểm Khó quản lý khi số lượng user lớn

2.2.2 VDSD - Mô hình ABAC (Attribute-Based Access Control)

Đặc điểm Mô tả
Cách thức phân quyền Tạo Policy (chính sách) dựa trên thuộc tính
Đơn vị quản lý Quyền chức năng + Quyền dữ liệu
Ưu điểm Linh hoạt, tự động áp dụng theo thuộc tính người dùng
Nhược điểm Phức tạp, khó debug khi có nhiều policy

Ví dụ Policy VDSD:

┌─────────────────────────────────────────────────────────────────────────────┐
│  VÍ DỤ POLICY VDSD                                                          │
└─────────────────────────────────────────────────────────────────────────────┘

  Policy: "Nhân viên chỉ xem được chỉ tiêu của đơn vị mình"
  
  ┌─────────────────────────────────────────────┐
  │  ĐỐI TƯỢNG ÁP DỤNG                          │
  │  ─────────────────                          │
  │  Loại: Đơn vị                               │
  │  Giá trị: TTPTDL (Trung tâm PTDL)           │
  └─────────────────────────────────────────────┘
                      │
                      ▼
  ┌─────────────────────────────────────────────┐
  │  ĐIỀU KIỆN                                  │
  │  ─────────                                  │
  │  Chỉ tiêu.đơn_vị_sở_hữu = User.đơn_vị       │
  │  HOẶC                                       │
  │  Chỉ tiêu.đơn_vị_sử_dụng = User.đơn_vị      │
  └─────────────────────────────────────────────┘
                      │
                      ▼
  ┌─────────────────────────────────────────────┐
  │  QUYỀN ĐƯỢC CẤP                             │
  │  ─────────────                              │
  │  ✓ Xem chỉ tiêu                             │
  │  ✗ Sửa chỉ tiêu                             │
  │  ✗ Xóa chỉ tiêu                             │
  └─────────────────────────────────────────────┘

2.2.3 Báo cáo VTM - Không có UI quản trị phân quyền

Đặc điểm Mô tả
Cách thức phân quyền Không có UI - Dev phải vào trực tiếp Database để cấu hình
Đơn vị quản lý Menu sidebar + iframe Tableau
Ưu điểm Đơn giản về mặt cấu trúc
Nhược điểm Phụ thuộc Dev, Admin không tự thao tác được, rủi ro khi thao tác trực tiếp DB

Luồng cấp quyền Báo cáo VTM:

┌─────────────────────────────────────────────────────────────────────────────┐
│  LUỒNG PHÂN QUYỀN BÁO CÁO VTM - KHÔNG CÓ UI                                   │
└─────────────────────────────────────────────────────────────────────────────┘

     ┌──────────────┐              ┌──────────────┐              ┌──────────────┐
     │    ADMIN     │              │     DEV      │              │   DATABASE   │
     │  Yêu cầu cấp │ ──────────▶ │  Nhận yêu   │ ──────────▶ │  INSERT/     │
     │  quyền cho   │              │  cầu, vào DB │              │  UPDATE trực │
     │  user X      │              │  thực hiện   │              │  tiếp        │
     └──────────────┘              └──────────────┘              └──────────────┘

  ❌ VẤN ĐỀ:
  • Admin không tự thao tác được, phải nhờ Dev
  • Rủi ro thao tác nhầm trực tiếp trên Database
  • Không có lịch sử thay đổi (audit log)
  • Phụ thuộc vào lịch của Dev

2.3 Luồng quản trị phân quyền hiện tại

Lưu ý: Người dùng đã có thể đăng nhập 1 lần qua SSO công ty. Vấn đề chính nằm ở phía Admin phải quản trị phân quyền ở nhiều nơi:

┌─────────────────────────────────────────────────────────────────────────────┐
│  LUỒNG QUẢN TRỊ PHÂN QUYỀN HIỆN TẠI                                         │
└─────────────────────────────────────────────────────────────────────────────┘

  Tình huống: Có nhân viên mới (Nguyễn A) cần được cấp quyền truy cập:

  ┌───────────────────────────────────────────────────────────────────────────┐
  │                                                                           │
  │  Bước 1 ──▶  Admin vào trang quản trị SQLWF                              │
  │              ┌─────────────────┐                                          │
  │              │ Admin SQLWF     │  → Tìm user Nguyễn A                     │
  │              │ Phân quyền      │  → Tick từng menu, từng permission       │
  │              └─────────────────┘  → ⏱ 15 phút                             │
  │                                                                           │
│  Bước 2 ──▶  Admin vào trang quản trị VDSD (UI khác hoàn toàn)           │
│              ┌─────────────────┐                                          │
│              │ Admin VDSD      │  → Tạo Policy cho Nguyễn A               │
  │              │ Tạo Policy      │  → Cấu hình điều kiện ABAC               │
  │              └─────────────────┘  → ⏱ 20 phút                             │
  │                                                                           │
  │  Bước 3 ──▶  VTM: Không có UI - Phải nhờ Dev                           │
  │              ┌─────────────────┐                                          │
  │              │ Admin gửi     │  → Yêu cầu Dev cấp quyền cho Nguyễn A    │
  │              │ yêu cầu Dev   │  → Dev vào DB thực hiện                   │
  │              └─────────────────┘  → ⏱ Phụ thuộc lịch Dev (có thể 1-2 ngày)  │
  │                                                                           │
  └───────────────────────────────────────────────────────────────────────────┘

  ❌ VẤN ĐỀ: 
  • Admin phải vào 2 hệ thống khác nhau (SQLWF, VDSD)
  • VTM không có UI - phải nhờ Dev, tốn thời gian chờ
  • Tổng: 35+ phút + chờ Dev (có thể 1-2 ngày)

3. CÁC VẤN ĐỀ VÀ THÁCH THỨC

3.1 Tổng hợp vấn đề

Lưu ý: Hệ thống SSO đã có sẵn - user chỉ cần đăng nhập 1 lần. Các vấn đề dưới đây tập trung vào việc quản trị phân quyền phân tán.

STT Vấn đề Mô tả Mức độ
1 Quản trị phân quyền phân tán Admin phải vào 3+ hệ thống khác nhau để cấp quyền cho 1 user 🔴 Nghiêm trọng
2 Giao diện quản trị không thống nhất Mỗi hệ thống có UI quản trị khác nhau, cách cấu hình khác nhau 🔴 Nghiêm trọng
3 Mô hình phân quyền khác nhau SQLWF dùng ACL, VDSD dùng ABAC, VTM dùng RBAC - khó thống nhất 🟡 Trung bình
4 Khó kiểm toán Không thể trả lời nhanh "User X có quyền gì trên toàn hệ thống?" 🔴 Nghiêm trọng
5 Tốn thời gian onboarding Mỗi user mới mất 40+ phút để cấu hình quyền đầy đủ 🔴 Nghiêm trọng
6 Chi phí bảo trì cao Khi cần thay đổi cơ chế phân quyền phải sửa nhiều nơi 🟡 Trung bình
7 Không có nhật ký tập trung Khó tra cứu lịch sử thay đổi quyền trên toàn bộ hệ thống 🔴 Nghiêm trọng

4. ĐỀ XUẤT GIẢI PHÁP (MONG MUỐN)

4.1 Tổng quan giải pháp

Xây dựng Hệ thống Phân quyền Tập trung (AuthCenter) - một giao diện quản trị duy nhất để cấu hình quyền cho tất cả các hệ thống:

STT Mục tiêu Mô tả
1 Giao diện quản trị tập trung Một Admin UI duy nhất để cấu hình quyền cho SQLWF, VDSD, Baocao.VTM
2 Profile quyền dùng chung Tạo các bộ quyền mẫu (Profile), gán nhanh cho user
3 Đa mô hình Hỗ trợ ACL, RBAC, ABAC trong cùng 1 hệ thống
4 Kiểm toán dễ dàng Tra cứu nhanh ai có quyền gì trên tất cả hệ thống
5 Tích hợp linh hoạt Các hệ thống con gọi API để kiểm tra quyền, linh hoạt trong việc mở rộng trong tương lai

Lưu ý: SSO đã có sẵn trong công ty. AuthCenter tập trung vào giải quyết vấn đề quản trị phân quyền phân tán.

PHỤ LỤC: BẢNG THUẬT NGỮ

Thuật ngữ Giải thích
ACL (Access Control List) Mô hình phân quyền cấp quyền trực tiếp cho từng user
RBAC (Role-Based Access Control) Mô hình phân quyền dựa trên vai trò (role)
ABAC (Attribute-Based Access Control) Mô hình phân quyền dựa trên thuộc tính (đơn vị, chức vụ...)
SSO (Single Sign-On) Đăng nhập 1 lần, truy cập được tất cả hệ thống
IdP (Identity Provider) Hệ thống xác thực danh tính (Keycloak, Auth0...)
Profile Nhóm quyền được định nghĩa sẵn để gán cho nhiều user
Policy Chính sách phân quyền, bao gồm điều kiện và hành động
Token (JWT) Mã xác thực điện tử chứa thông tin user, dùng để truy cập hệ thống
SDK Bộ công cụ phát triển giúp tích hợp hệ thống với AuthCenter
Audit Log Nhật ký ghi lại các hoạt động để kiểm toán
